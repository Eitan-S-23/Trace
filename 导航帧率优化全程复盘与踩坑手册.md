## 一、最终架构(当前代码状态)

数据通路:GPS(模拟器/真实)→ 显示坐标插值 → 视口快照(RAM)→ LVGL 直拷 → SPI 屏。

| 组件 | 位置 | 要点 |
|---|---|---|
| SDIO 多块读 | `MDK-ARM_F435/Platform/Core/at32_sdio.c`、`Libraries/SdFat/src/SdCard/SdioCard_AT32.cpp` | CMD18 + **同步** CMD12(数据完成后在函数内发,不依赖中断);失败降级逐块 |
| 瓦片行缓存 | `Simulator/.../lv_img_decoder.c`(共享文件,宏仅 ARDUINO 生效) | 3 条 × 16 行(8KB/条),`XTRACK_IMG_LINE_CACHE_*` |
| GPS 插值 | `LiveMap.cpp CheckPosition` | 显示坐标每周期向 GPS 目标指数逼近(剩余 1/3,≤1px 吸附);轨迹记录用真实坐标 |
| 扩展 SRAM | `startup_at32f435_437.s` → `extend_sram_512k()`;`MDK-ARM_F435/Objects/X-Track.sct` | EOPB0 配 512K;快照区 `RW_IRAM2 0x20058000 UNINIT 0x28000` |
| 视口快照 | `LiveMap.cpp`(`Snapshot_*` 函数族)、`LiveMapView.cpp Map_Create` | 256×320×2=160KB 常驻 `.sram_ext`;滚动 memmove+边条读;`CONFIG_LIVE_MAP_SNAPSHOT_ENABLE` |
| RTT 调试命令 | `App.cpp`(`CONFIG_RTT_DEBUG_CMD_ENABLE`) | `ping/livemap/dialplate/back/gpsreset`;生产置 0 整体移除 |

配置开关(`Config.h`):`SNAPSHOT_ENABLE=1` 与 `RECENTER_ENABLE=0` 互斥;`SCROLL_INTERP_*` 插值参数。

## 二、性能账本(全部设备实测,J-Link+RTT)

### 优化演进(400km/h 压测工况,引擎吃满)

| 阶段 | 整屏帧率 | 每帧构成 | commit |
|---|---|---|---|
| 原始(逐块读+8行缓存) | ~5FPS(日常)/13.3(压测) | 76ms = SD 42 + 渲染 34 | — |
| SDIO 多块读修复 | 13.3 → 单帧 SD 减半 | 单次 4KB miss 1.30→0.53ms | `5f22583` |
| 行缓存 16 行/条 | **15.4FPS** | 65ms = SD 31 + 渲染 34 | `9a2b8bf` |
| GPS 插值 | (解除 5Hz 步进,不改能力) | — | `1726843` |
| **视口快照** | **24.5FPS** | **19.4ms = SD 1.0 + 渲染 18.4** | `7d48f02` |

### 帧率的三层限制(重要认知,勿再混淆)

实际帧率 = min(**像素速度**, 刷新节流 31Hz, 渲染能力 24.5+):

- **像素速度** = 物理速度 ÷ 地图分辨率。zoom16@65km/h ≈ 8px/s → 日常帧率 5-8FPS **是内容变化率所限,不是性能问题**(引擎空闲 89%)。位图平移最小粒度 1px,超过像素速度的刷新全是重复帧。
- 插值突破的是 **GPS 采样率**(5Hz→像素速度);1px 粒度理论可用亚像素重采样突破,但 M4 无 GPU,每帧全屏双线性 ~40ms 不可行。
- 想看到"高帧率数字":提高 zoom(17/18)或速度;快照优化买到的是"任何缩放/速度都顶到天花板"。

### 单项关键数字

- SDIO 单块事务固定开销 ~150µs(命令+NAC+FAT,与数据量无关)→ 一切"减少事务数"的优化都有效,"提时钟"无效(48MHz 已顶格,数据仅占 7%)。
- 行缓存 LRU 悬崖:缓存 < 整帧工作集(~40 条)时跨帧命中率≈0,**扩条数是伪优化**;加大**读粒度**(8→16 行)摊薄固定开销才有效;32 行/条实测增益反转(勿再加大)。
- AA 线条(GPX 路线)≈12ms/整屏帧,仍每帧重画(未烘焙进快照,后续可选)。
- 整屏 SPI 传输 17ms(72MHz 顶格)与渲染并行;REFR_PERIOD 32ms 是当前软上限,降 16ms 可冲 ~50FPS(引擎余量足)。

## 三、踩坑手册(症状 → 根因 → 修法/预防)

### 坑 1:启用 CMD18 多块读后 SD 全挂(白屏、sdMs=0)

- **根因**:本工程 **SDIO 的 NVIC 中断从未使能**,`sd_irq_service`(含 stop_flag=1 时中断内发 CMD12)是死代码。多块读数据收完没人停卡 → 卡停在 data 态 → 后续所有命令失败。单块读不需要停卡,故多年未暴露。
- **修法**:`sd_mult_blocks_read` 置 `stop_flag=0`,数据完成后**同步**发 CMD12(commit `5f22583`)。
- **预防**:改驱动前先确认中断链路是否真的在跑(NVIC enable + 向量表);任何"首次启用"的硬件路径必须先做**带隔离的自检实验**(参考 `HAL_SD_CARD.cpp` 的 `CONFIG_SD_MULTIBLOCK_SELFTEST`:开机测 A/B/C 三路径、RTT 报错误码、随后完整重新初始化,功能不受影响)。

### 坑 2:砍掉热点 CMD16 后挂载失败(第一次白屏事故)

- **根因**:`sd_init` 内 `scr_find`(块长设 8)/`sd_switch`(设 64)会改卡块长,旧代码靠**每次读写前的 CMD16(512)** 掩盖;跳过热点 CMD16 后首个 512B 读撞上残留块长。
- **修法**:`sd_init` 末尾一次性发 CMD16(512)(全局前置条件,非逐次开销)。
- **教训**:删"冗余"操作前,先确认它是否在无意中维持着某个全局不变量。

### 坑 3:烧录导致 SD 卡挂死(J-Link halt 打断 SDIO 传输)

- **症状**:`SD_IsReady=0`、stat 的 miss/sdMs 全 0、瓦片消失;软复位救不回。
- **修法**:拔插 SD 卡(CD 引脚自动重挂载)或整机断电。
- **预防**:误判前先从 map 查 `SD_IsReady` 地址用 J-Link `mem` 读值确认,别当成代码 bug 追。

### 坑 4:快照水平边条读放大 60 倍(4MB/s 废读拖死主循环)

- **根因**:瓦片是行主序位图,水平滚动的 1-4px 窄条要逐行 ReadLine、纵向跨 20 个 band,每 band 8KB 全读只用 128B。
- **修法**:快照加 16px 水平 margin(256 宽)、原点按 16px 网格对齐——滚动频率与放大同降一个量级。垂直滚动读整行,无此问题。
- **通用规律**:对行主序位图做**窄列**访问必然读放大,设计时让高频访问方向与存储主序一致。

### 坑 5:缺瓦片区域把主循环拖到 1/4 速

- **根因**:每行重试打开不存在的文件 = 每次一趟 FAT 全目录遍历(比读数据贵)。
- **修法**:失败瓦片 2 秒负缓存(`snapFailKey[4]`)。
- **通用规律**:失败路径往往比成功路径更贵,负缓存是标配。

### 坑 6:快照模式整页白屏(容器高度 0,最阴险的一个)

- **根因**:`LiveMapView::SetMapTile` 按 `tileNum/widthCnt` 算容器高度,快照模式 `tileNum=0` → 容器 512×**0** → 快照/箭头/线条全部被 LVGL 可见性裁剪。**三个症状同源**:白屏、箭头消失、`lv_obj_invalidate(img)` 无效(与绘制走同一套裁剪)。
- **修法**:快照模式容器尺寸直接取瓦片容器矩形(commit `4879137`)。
- **判别手法**(可复用):模拟器上把失败填充色临时改红→截图:红=img 在画(内容问题);白=img 没被画(裁剪/可见性问题)。**模拟器 PC 路径下有地图数据,可完整验证显示链路**。
- **教训**:被复用的函数对"空集合"参数的行为要检查(除以 0、尺寸 0);LVGL 对象不显示时优先查父链尺寸与裁剪,再查内容。

### 坑 7:GPS 模拟器跑出地图区(反复干扰测量的头号环境陷阱)

- **机制**:模拟器 50-80km/h 随机游走,几十分钟必出 SD 卡瓦片覆盖区;且经轨迹 GPX 的 `LoadLastPosition` **持久化**出界位置,复位也回不来。
- **症状**:miss/sdMs 全 0 + 地图空白 + 帧率虚高(内容稀疏),曾三次被误判为代码 bug。
- **修法**:RTT 命令 `gpsreset` 归位默认坐标(104.88,26.57 六盘水,地图覆盖区内)。**测量标准流程:复位 → gpsreset → livemap 三连**。

### 坑 8:测量边界陷阱("渲染慢"的历史误判)

- 本架构 SD 读**嵌在渲染管线内部**(draw→read_line→miss→SD),测"渲染耗时"必然把 SD 等待计入 → 早期文档"瓶颈在渲染调用"的归因错误由此而来。
- **修法**:用 DWT 在 miss 分支单独记账(`sdMs`),`refrMs - sdMs` 才是纯渲染。已固化在 stat 输出里。
- **通用规律**:profiling 先想清楚测量边界包含了什么等待;"耗时高的函数"未必是根因。

### 坑 9:定点坐标 int32 溢出(仅高级别空白,指纹极具迷惑性)

- **症状**:16/17/18 级整页空白,**15 级及以下完全正常**——看起来像"级别相关的数据/路径问题",实为算术溢出。
- **根因**:插值定点化(commit b28a032)用 int32 存 `mapX << 8`:16 级像素坐标 ≈1.3e7,左移后 3.4e9 超 int32 上限溢出为负 → 显示坐标变成 -350 万 → 快照读取全部落入"负坐标填白"分支。15 级坐标 6.6e6<<8=1.7e9 恰在溢出线下。
- **修法**:`dispMapXFp/YFp` 升 int64(commit 30dab34)。
- **通用规律**:**"级别 N 以上坏、N 以下好"= 坐标幅值分界 = 溢出指纹**,先查算术位宽再查数据;像素坐标(19 级可达 1.3e8)做任何定点左移前必须核对幅值上限。
- **定位手法**:模拟器 fopen 取证打印坐标即水落石出(gx=-3500672 与推算值精确吻合)。

### 坑 10:亚像素实验——"理论预算"与实测差 5 倍

- **症状**:预估 LVGL 变换路径 +13ms/帧,实测 **~70ms/帧**(refrMs 1030/s 吃满,帧率反降至 12FPS)。
- **原因**:LVGL8 软件变换是通用路径(任意角度矩阵、逐像素函数调用、Flash 取指),不是为"纯平移相位"特化的。
- **通用规律**:凡"首次启用的重型库路径",纸面周期数估算不可信,**必须先做性能探针实验**(最小代码量测真实成本)再决定实施——这与坑 1 的"自检先行"是同一纪律在性能域的版本。
- 路线三(decoder 层相位混合)已在 2026-07-10 clean retry 阶段 0 探针否决:read_line memcpy A=26.10ms/帧,固定半像素双线性 B=101.81ms/帧,`refrMs=1015.3ms/s`,`update=9.97`,远超 >35ms 终止线。根因不是 SD(`sdMs=6.62ms/s`),而是每帧 240x320 双线性+LVGL 逐行解码在 M4/AC5 上吃满 CPU。
- 完整实验记录与三条技术路线的评估见 `PLAN-SUBPIXEL.md`。

## 四、测量与自动化方法(已固化,直接照用)

J-Link 闭环(烧录/RTT/命令/复位)详见 `AGENTS.md` "J-Link 自动烧录与 RTT 闭环调试"一节,此处只补充本轮新增:

1. **快照像素级验证**:`savebin` dump `snapshotBuf`(map 查地址)→ Python RGB565 转 PNG 直接看内容。生产侧(快照内容)与显示侧(LVGL 绘制)可分别验证。
2. **stat 行判读口诀**:`sdMs/refrMs` 高=SD 主导;`refrMs→1000`=引擎吃满;`refrPxK/76.8`=每秒整屏数;`update` 掉到 20 以下=有东西拖死主循环;miss/readKB 全 0 且"应该有图"=先怀疑出界或 SD 挂死,不是渲染 bug。
3. **压测方法**:临时把 `SIM_SPEED_MIN/MAX_KPH` 调到 300-400(记得测完恢复 50-80),使引擎 100% 吃满测能力上限;日常速度测的是内容变化率不是能力。
4. **每次烧录后必须从 map 重查 `_SEGGER_RTT` 地址**(bss 布局会漂移),down buffer 描述符随之重读——忘记这条曾浪费一轮采集。严格做法是用 `_SEGGER_RTT` 符号行解析,再用 `mem8 <RTT> 16` 确认 `SEGGER RTT` 签名,不要抓上下文里的第一个十六进制数。
5. **RTT logger 只能有一个**:采集前先 `Stop-Process -Name JLinkRTTLogger -Force -ErrorAction SilentlyContinue`,采集必须有超时并在结束后确认无残留。残留 logger 会抢 RTT 单读指针,让日志缺行、串行或混入旧固件输出。
6. **down channel 是环形缓冲**:连续发送 `gpsreset`/`livemap` 必须按 descriptor 的 WrOff/Size/RdOff 取模续写;若回显 `unknown 'psreset'` 等缺首字母命令,说明写偏移或 descriptor 已错,当前日志作废,复位并重读 descriptor 后重测。
7. 一次只变一个变量;两个观测矛盾时先取证(加临时打印/dump)再修,禁止连续盲改(SDIO 白屏事故 = 连续盲打补丁的反面教材)。

## 五、遗留事项与后续方向

- **平滑度(非帧率)是当前唯一显著的体验短板**:30km/h@zoom16 画面每秒仅移动
  ~4 像素,1px@4Hz 的跳格感 ≠ 帧率不足(引擎空闲 84%),提高刷新率无效。
  解法 = 亚像素渲染;LVGL 变换路线、独立显示缓冲路线、decoder 层相位混合
  路线均已否决(坑 10)——完整计划见 `PLAN-SUBPIXEL.md`。
- **可选**:`CONFIG_LIVE_MAP_REFR_PERIOD` 32→16ms 冲 ~50FPS(仅高倍缩放/高速
  场景有意义);路线/轨迹线烘焙进快照(消掉 12ms/帧的 AA 线条重画)。
- **实验残留**(判读完可关):`CONFIG_SD_MULTIBLOCK_SELFTEST`(已置 0)、stat
  统计代码(保留作长期标尺)、`CONFIG_RTT_DEBUG_CMD_ENABLE`(生产置 0)。
- 写路径仍逐块(CMD25 未启用),SystemSave/轨迹写入量小,暂无必要。
