# 导航界面（LiveMap）帧率优化方案

**生成时间**：2026-06-15
**适用平台**：AT32F435RGT7（系统时钟 288MHz，SRAM 链接区 384KB）
**问题现象**：设备信息界面（SystemInfos）稳定 60 FPS，导航界面（LiveMap）仅约 15 FPS。

> 本文档基于对真实 F435 工程（非模拟器）源码的逐层走查，结论与既有的
> `性能优化总结.md`、`LiveMap渲染流程技术文档.md` **不同**：那两份文档把瓶颈归因于
> `lv_obj_set_pos()` / `lv_img_set_angle()` 的无意义调用，并据此做了位置/角度缓存优化，
> 但实测帧率仍为 15 FPS，说明真正的瓶颈未被定位。本文给出经证据链确认的根因与方案。

---

## 一、为什么两个界面帧率差距如此之大（根因）

两个界面跑的是同一套 LVGL 渲染引擎、同一块屏幕、同一条 SPI+DMA 通路。
帧率差异的本质是**每帧需要重绘的"脏区域面积"与"像素数据来源"完全不同**。

### 1.1 设备信息界面：只重绘极小的局部

SystemInfos 是基本静止的页面，只有少量文本标签会变化。LVGL 的脏区域机制
（`lv_refr.c`）只会重绘发生变化的小矩形（几十像素宽的标签），每帧通过 SPI
发送的数据量极小，且像素全部来自 RAM 中的字体位图。因此轻松达到刷新周期上限
60 FPS（`lv_conf.h:90` `LV_DISP_DEF_REFR_PERIOD 16`）。

### 1.2 导航界面：每帧重绘整屏 + 像素来自 SD 卡

- 地图容器 `ui.map.cont` 覆盖整个可视区（`Config.h:81-82`
  `CONFIG_LIVE_MAP_VIEW_WIDTH/HEIGHT = LV_HOR_RES/LV_VER_RES = 240×240`）。
- 每次 GPS 位置变化，`LiveMap::MapTileContUpdate()` 都会调用
  `View.SetMapContPos()` 移动整个地图容器（`LiveMap.cpp:408-433`）。容器一移动，
  LVGL 就把"旧位置+新位置"标记为脏 → **几乎整屏 240×240 失效**。
- 运动状态下位置持续变化，于是**每一帧都是整屏重绘**。

这一条已经决定了导航界面比设备信息界面重得多。但真正把帧时间拉到 ~66ms 的，是
下面这个被忽略的事实 ⬇️。

### 1.3 致命瓶颈：地图瓦片像素每帧从 SD 卡逐行流式读取

LVGL 堆只有 72KB（`lv_conf.h:61` `LV_MEM_SIZE (72U*1024U)`），
而单张瓦片是 `256×256×2 = 128KB`，**根本放不进堆**。因此工程给 LVGL 内置解码器
打了补丁，采用"逐行读取 + 小行缓存"的方式，从 SD 卡按需读取瓦片像素：

- 实现位置：`Simulator/LVGL.Simulator/lvgl/src/draw/lv_img_decoder.c`
  （F435 工程通过包含路径复用了 `Simulator/.../lv_conf.h` 与该补丁文件）。
- 硬件上启用：`lv_img_decoder.c:21-30`，`#ifdef ARDUINO` 时
  `XTRACK_IMG_LINE_CACHE_ENABLE 1`。
- 行缓存规模：`XTRACK_IMG_LINE_CACHE_CNT 6` × `XTRACK_IMG_LINE_CACHE_ROWS 8` ×
  `256宽 × 2字节` = **仅 24KB（6 条、共 48 行）**。
- 未命中处理：`xtrack_img_line_cache_read()`（`lv_img_decoder.c:595-684`）
  对该瓦片文件 `lv_fs_seek()` + `lv_fs_read()`，从 SD 卡读 8 行（4KB）。

**问题在于缓存太小，撑不住一帧的整屏渲染：**

- 整屏 240 行、横向通常跨 2 张瓦片。LVGL 自顶向下扫描渲染，每个 8 行带需要
  每张相关瓦片各读一次。一帧需要触碰的"瓦片 8 行带"约 `240/8 × 2列 ≈ 60` 个。
- 但行缓存只有 6 条（48 行）。扫到屏幕下半部分时，上半部分的缓存条早被淘汰；
  下一帧从头扫，又得**全部重新从 SD 卡读**。缓存命中率极低 → 缓存抖动（thrashing）。
- 即每帧约 `60 × 4KB ≈ 240KB` 的 SD 随机读。SDIO 4KB 随机读（含命令/寻址开销）
  约 0.5–1ms/次，`60 × ~0.7ms ≈ 42ms`，**单这一项就吃满了 15 FPS 的帧预算**。

工程其实已埋了观测点：`LiveMap.cpp:207-233` 每秒通过 RTT 打印
`lineHit / lineMiss / lineReadKB`。**运行时若看到 lineMiss 很高、lineReadKB 达数 MB/s，
即可实锤本瓶颈。**

### 1.4 叠加的次要放大因素

在"整屏重绘 + 逐行读 SD"的基础上，以下因素进一步拖慢每帧：

1. **`LV_COLOR_SCREEN_TRANSP 1`**（`lv_conf.h:40`）。开启屏幕级透明会让 LVGL 走
   更慢的逐像素 alpha 混合路径。`LiveMap渲染流程技术文档.md` 自己也写明此项应为 0
   可带来 3–4 倍提升，但实际配置仍是 1。该开销按"整屏像素"计，因此专门放大导航界面。
2. **半透明叠加面板 + 阴影**。`LiveMapView.cpp:54-59` 的 `styleCont`：
   `bg_opa = LV_OPA_60`、`shadow_width = 10`。速度信息面板与缩放面板用了该样式
   （`LiveMapView.cpp:215, 178`）。阴影渲染是 LVGL 最昂贵的操作之一，而面板下方的
   地图每帧都在变 → 阴影与半透明混合**每帧重算**。
3. **抗锯齿粗轨迹线**：`styleLine` 宽 5、圆角、`LV_OPA_COVER`
   （`LiveMapView.cpp:65-69`），多段 `lv_line` 抗锯齿绘制有一定成本。
4. **箭头旋转**：`lv_img_set_angle()` 软件旋转，每次约数 ms（已有 1° 阈值缓解，
   但运动转向时仍会触发）。
5. **整屏 SPI 传输**：整屏 `240×240×2 = 115KB`，分两个半屏缓冲发送。
   SPI 实际时钟约 72MHz（见下），传输约 13ms——显著但不是最大头。

### 1.5 顺带澄清：显示 SPI 并非瓶颈，无需"提速"

- `Adafruit_ST7789::begin()`（`Adafruit_ST7789.cpp:73-74`）调用
  `spi->setClock(F_CPU)`，`F_CPU = system_core_clock = 288MHz`。
- `SPIClass::setClock()`（`SPI.cpp:175-214`）：`clockDiv = 288M/288M = 1`，
  映射为 `SPI_MCLK_DIV_2`。SPI1 在 APB2（`at32f435_437_clock.c:85` `APB2_DIV_2` → 144MHz），
  故显示 SPI ≈ **72MHz**，已接近 ST7789 上限。**SPI 没有可压榨空间**，
  早期"怀疑 SPI 速度"的方向是错的（用户当时也已否定）。

## 二、优化方案（按性价比分级）

> 总策略：**先用零风险的配置项拿回大头，再削减整屏重绘的频率/面积，
> 最后从结构上消灭"每帧读 SD"。** 每一步都用 1.3 节的 RTT 统计验证效果，避免拍脑袋。

### P0　配置级快速止血（改动小、风险低、立刻见效）

| 项 | 位置 | 改动 | 预期 |
|----|------|------|------|
| 关闭屏幕透明 | `lv_conf.h:40` | `LV_COLOR_SCREEN_TRANSP 1 → 0` | 整屏混合走快速路径，3–4 倍混合提速 |
| 去掉面板阴影 | `LiveMapView.cpp:58-59` | `shadow_width 10 → 0`（或移除 shadow 样式） | 消除每帧昂贵的阴影重算 |
| 面板改不透明 | `LiveMapView.cpp:56` | `bg_opa LV_OPA_60 → LV_OPA_COVER` | 面板区域跳过 alpha 混合（视觉略变，可权衡） |

> 注意：`LV_COLOR_SCREEN_TRANSP` 改 0 后需确认根屏幕不依赖透明背景
> （LiveMap 根对象在 `LiveMapView.cpp:19` 设为白色不透明背景，满足条件）。

### P1　削减整屏重绘的频率与面积（中等改动）

1. **地图容器移动加位移阈值/累积**
   - 现状：`CheckPosition()` 中只要 `mapX/mapY` 变化就触发
     `MapTileContUpdate()` 整屏移动（`LiveMap.cpp:346-374`）。
   - 改法：累积位移，仅当 `|Δ| ≥ N` 像素（如 2–4px）才真正移动容器并整屏重绘；
     不足阈值时只更新箭头（小脏区）。直接降低整屏重绘频次。
2. **降低地图刷新频率**
   - `Config.h:73` `CONFIG_LIVE_MAP_REFR_PERIOD 30ms`（≈33Hz）。可调到
     50–66ms（15–20Hz）。地图平移到 20Hz 视觉已足够顺滑，CPU/SD 负载下降约 1/3。
3. **让叠加面板脱离地图重绘区**
   - 面板（速度信息/缩放）覆盖在每帧重绘的地图之上，导致其混合每帧重算。
     P0 改不透明后已大幅缓解；进一步可考虑把面板放到独立图层或缩小覆盖面积。

### P2　结构性根治：消灭"每帧读 SD"（核心、收益最大）

目标：把瓦片像素读取从"每帧 ~240KB"降到"仅在瓦片切换时读一次"。两条路线，二选一：

- **路线 A（推荐，改动集中）：增大瓦片行缓存，使其覆盖整屏一帧所触碰的全部 8 行带。**
  - 位置：`lv_img_decoder.c:22-27`，调大 `XTRACK_IMG_LINE_CACHE_CNT`。
  - 命中逻辑按 `(文件hash, y_start, width, row_cnt)` 匹配（`:637-644`），缓存够大时，
    静止/慢速场景下第二帧起几乎全命中，只有滚动边缘新露出的少数带才未命中。
  - 容量测算：当前链接区 384KB（`X-Track.sct:12` `RW_IRAM1 0x20000000 0x00060000`），
    已用约 282KB（map：`Total RW Size 288944`），**空余约 100KB**。
    一帧约需 60 条 × 4KB ≈ 248KB，**当前空余装不下整帧**。因此：
    - 先把 `CNT` 从 6 提到上限（约 18 条 / 72KB），覆盖约 144 行，已能把命中率和
      lineReadKB 显著拉低；用 RTT 统计验证增益。
    - 要完全覆盖整帧，需配合下方"扩 SRAM"。
- **路线 B（更彻底）：按"瓦片容器变化"整张预解码到专用 RAM 池。**
  - 在 `onMapTileContRefresh()`（`LiveMap.cpp:386-406`，仅在瓦片真正切换时触发）
    把当前可见的瓦片整张读入一块专用瓦片 RAM；渲染时解码器直接从该 RAM 取像素，
    彻底取消渲染期的 SD 访问。
  - 代价：需要瓦片 RAM。可只缓存"可见窗口"而非 4 整张：把合成后的 240×240 地图位图
    缓存为单张（`240×240×2 = 115KB`），瓦片不变时每帧仅平移该位图。

**扩大可用 SRAM（路线 A/B 的使能前提）**
- AT32F435RGT7 物理 SRAM 大于当前链接所用的 384KB，但 `X-Track.sct` 只声明了
  `0x00060000`。可将 `RW_IRAM1` 区域扩到 `0x00080000`（512KB，并按数据手册配置
  扩展 SRAM 容量寄存器），释放出 ~128KB 给瓦片缓存/行缓存，使整帧驻留成为可能。
- 该项需核对具体料号 SRAM 容量与 SRAM 大小配置位后再启用。

### P3　渲染/传输并行与传输优化（锦上添花）

- 当前 `disp_wait_cb` 用 `__wfi()`（`lv_port_disp.cpp:27-30`）阻塞等待 DMA。
  双半屏缓冲下 LVGL 已能在 DMA 发送上半屏时渲染下半屏，存在部分重叠；确认
  `lv_disp_flush_ready` 在 DMA 完成中断里调用（`HAL_Display.cpp:93-96` 已是）。
- DMA 目前按**字节宽**传输（`HAL_Display.cpp:113-116`
  `DATA_WIDTH_BYTE`）。RGB565 是 16 位，可评估改为半字宽 + 16 位 SPI 帧，
  减少一半 DMA 请求次数（需同步改 SPI `frame_bit_num` 与字节序），属微调。

## 三、实施与验证顺序（强制本地验证）

1. **先量化基线**：在 LiveMap 运行态读取 RTT（`LiveMap.cpp:218-226`），记录
   `update / reload / lineHit / lineMiss / lineReadKB`。预期看到高 lineMiss 与
   数 MB/s 的 lineReadKB —— 实锤 1.3 节根因。
2. **做 P0**：改 `LV_COLOR_SCREEN_TRANSP=0`、去阴影、面板不透明。重新测 FPS。
   预期静止态明显提升、移动态有改善。
3. **做 P1**：加位移阈值 + 降地图刷新频率。再测移动态 FPS 与 lineReadKB。
4. **做 P2 路线 A**：增大行缓存 `CNT`，对照 lineHit/lineMiss/lineReadKB 调参到收益拐点；
   若空余 RAM 不足，先扩 SRAM 链接区再加大。
5. 若仍不达标，评估 P2 路线 B（可见窗口位图缓存）与 P3。

每步均以"FPS + RTT 统计"对照前后数据留痕，任一步无正向收益则回退该步并复盘。

## 四、预期结果

| 阶段 | 关键改动 | 预期导航帧率 |
|------|----------|--------------|
| 基线 | — | ~15 FPS |
| P0   | 透明关闭 + 去阴影 + 面板不透明 | 25–35 FPS |
| P0+P1 | 叠加位移阈值 + 降刷新频率 | 30–45 FPS |
| P0+P1+P2 | 消灭每帧读 SD（缓存整帧瓦片像素） | 45–60 FPS（接近设备信息界面） |

**一句话结论**：导航界面慢，不是显示硬件或 `lv_obj_set_pos` 的问题，而是
**"每帧整屏重绘"叠加"瓦片像素每帧从 SD 卡逐行重读"**——前者由地图容器随 GPS 移动
导致，后者由 72KB 堆放不下瓦片、行缓存仅 24KB 撑不住整帧导致。优先用 P0 配置项止血，
再用 P2 让瓦片像素常驻 RAM 根治。

## 六、已实施改动与验证（2026-07-02）

本轮基于上述分析落地了**零 RAM 风险**的核心方案，并对行缓存方向做了修正。

### 6.1 采纳"recenter + 平滑归位"，放弃"适度扩行缓存"

对 `lv_img_decoder.c:619` 逐行走查后确认：行缓存 key 是 **8 行瓦片对齐**
`y_start = (y/8)*8`，命中判定 `(文件hash, y_start, 宽, 行数, px_bytes)`。据此推演：

- **单帧整屏重绘**内部是自顶向下单调扫描，每个 8 行带只读一次，**与缓存大小无关**。
- **跨帧复用**才是缓存大小唯一起作用处。但 LRU 存在"悬崖"：
  - 缓存 **< 整帧工作集**（视口约 `(高/8)×2列 ≈ 60~80 条 = 240~320KB`）→ 帧顶部 band
    在扫到底部前必被挤出，下一帧从顶重扫又全量 miss → 跨帧命中率 **≈ 0**。
  - 缓存 **≥ 整帧工作集** → 跨帧命中率 **≈ 100%**。
- 结论：`CNT` 6→12（48KB）远在悬崖下方（12 ≪ 60~80），**帧率不会改善**；要越过悬崖需
  ~240~320KB，当前仅余 ~92KB，必须先扩 SRAM 到 512KB（高风险）。因此**第 §2 P2-A 的
  "适度扩行缓存"被判定为伪收益，本轮不做**。

真正让"每帧读 SD"消失且零 RAM 代价的是 recenter：地图静止后每帧只重绘箭头 ~32×32 小区，
其下 4~8 条 band 每帧相同 → **现有 6 条缓存即 100% 命中**，SD 读趋近 0。

### 6.2 落地清单

| 文件 | 改动 | 作用 |
|------|------|------|
| `Config.h` | `CONFIG_LIVE_MAP_RECENTER_ENABLE 0→1`；新增 `RECENTER_ANIM_DIV=4`、`RECENTER_ANIM_SNAP=3` | 启用 recenter，并把硬跳改为平滑归位 |
| `LiveMap.cpp::MapTileContUpdate` | 硬跳 → 指数缓动（每周期逼近剩余 1/DIV，≤SNAP 吸附收尾） | 消除"瞬间跳"，观感平滑 |
| `LiveMap.cpp::CheckPosition` | 归位中（`recentering`）绕过提前返回 | GPS 静止时动画也能走完 |
| `LiveMap.h` / `onViewDidAppear` | 新增并复位 `bool recentering` | 归位动画状态机 |
| `LiveMapView.cpp::Style_Create` | 删除 `styleCont` 的 `shadow_width=10` / `shadow_color` | 去掉每帧最贵的阴影重算（且符合工程禁用 shadow_* 规则） |

未采纳项（供后续权衡）：面板改不透明（§2 P0）、降刷新频率（§2 P1）、扩 SRAM+整帧瓦片
常驻（§2 P2-B / 悬崖之上的行缓存）。若日后要**保持平滑滚动同时高帧率**，只能走扩 SRAM 路线。

  （注：模拟器用 PC malloc + PC 文件系统，**不复现** SD 逐行读瓶颈，仅验证编译与渲染无破坏。）
- 固件：`build_f435.ps1 -AutoStale`，`armlink`/`fromelf` 退出码 0，无报错。
  - `Program Size: Code=261792 RO-data=288232 RW-data=1200 ZI-data=297708`
  - RW+ZI = 298908 字节 ≈ 291.9KB，**与改动前一致**（仅多 1 个 bool），384KB 区仍余 ~92KB。
  - 输出：`X-Track.axf/.hex`、`Track.bin`，时间戳 2026-07-02 17:56。

### 6.4 待设备端确认（本机无法烧录）

烧录 `Track.bin` / `X-Track.hex` 进导航页，读 `LiveMap.cpp` 每秒 RTT
`update / reload / lineHit / lineMiss / lineReadKB`：
