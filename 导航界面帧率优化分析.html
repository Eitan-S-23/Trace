导航界面（LiveMap）帧率优化分析

本文档记录 X-Track 导航界面帧率偏低问题的完整排查过程：从"设备信息界面 60 FPS、 导航界面仅 ~15 FPS"的现象出发，经源码走查与 RTT 实测数据，定位真正瓶颈， 验证若干优化手段的实际收益，并给出已实施改动与后续方向。

AT32F435RGT7 · 288MHz LVGL v8 ST7789 240×320 SPI + DMA 显示 SDIO 4-bit 地图瓦片 每帧整屏重绘 周期性 recenter
平台: AT32F435RGT7（系统时钟 288MHz，SRAM 链接区 384KB）  ·  显示: ST7789 240×320，SPI1 @ ~72MHz + DMA  ·  地图: SDIO 4-bit 读取 256×256 RGB565 瓦片
核心结论（先说）：导航界面慢的本质是 "为几像素的移动付整屏重绘的全价"——地图容器随 GPS 每帧移动 → 整屏失效 → 整屏渲染 + 整屏 SPI 传输 + 瓦片像素每帧从 SD 重读。 SD 提速、关闭屏幕透明等"局部优化"都绕不开这个前提，因此只涨一两帧； 唯一的结构性突破是让大多数帧不再整屏重绘（周期性 recenter）。

同一块屏、同一条 SPI，设备信息界面能跑 60 FPS——这证明显示硬件没问题， 问题出在"导航界面每帧要重绘的面积"远大于"设备信息界面只重绘几个文字标签"。

现象与早期误判

两个界面跑同一套 LVGL 渲染引擎、同一块屏幕、同一条 SPI+DMA 通路，帧率却相差 4 倍：

界面每帧重绘内容像素数据来源帧率
设备信息 SystemInfos仅变化的小文字标签（小脏区）RAM 字体位图60 FPS
导航 LiveMap整屏地图 + 轨迹 + 箭头SD 卡瓦片像素~15 FPS
被否定的旧结论：既有的 性能优化总结.md 把瓶颈归因于 lv_obj_set_pos() / lv_img_set_angle() 的无意义调用， 并据此加了位置/角度缓存。但实测帧率仍是 ~15 FPS，说明该方向没命中真正瓶颈。

根因分析

① 每帧整屏重绘

地图容器 ui.map.cont 覆盖整个可视区（CONFIG_LIVE_MAP_VIEW = LV_HOR_RES × LV_VER_RES）。 每次 GPS 位置变化，MapTileContUpdate() 都会移动整个容器，让箭头锁定屏幕中心、地图在底下平滑滚动。 容器覆盖全屏，一移动 → LVGL 把旧位置+新位置标记为脏 → 几乎整屏 240×320 失效重绘。 运动状态下位置持续变化，于是每一帧都是整屏重绘。

② 瓦片像素每帧从 SD 卡逐行读取

LVGL 内置文件解码器对"文件 + 真彩色"瓦片本就是逐行流式读取的设计： lv_img_decoder_built_in_open() 只开着文件就返回，img_data 保持 NULL， 绘制时由 read_line 回调按行从 SD 取像素（见 lv_img_decoder.c:444-447）。 工程在此之上加了"小行缓存"补丁：硬件上 XTRACK_IMG_LINE_CACHE 仅 6 条 × 8 行 × 256 宽 × 2 字节 = 24KB（一个独立的 static 数组，不在 LVGL 堆里）。 整屏渲染需要触碰约 60 个"瓦片 8 行带"，缓存撑不住一帧 → 缓存抖动 → 每帧约 100~240KB 从 SD 重读。

误区澄清：并非"LVGL 堆（72KB）放不下 128KB 瓦片才被迫流式"。真相是解码器无论堆多大都逐行流式， 压根不会整图载入。因此单纯增大 LV_MEM_SIZE 对帧率零影响，只会白吃掉本就紧张的 ~100KB 空余 RAM。 详见"帧率天花板"一节。

③ 三块成本都正比于"像素数量"

渲染时间 ∝ 像素数  ·  SD 读取 ∝ 像素数（覆盖多少瓦片行）  ·  SPI 传输 ∝ 像素数
所以"减面积"能同时砍掉三块成本，而单独优化某一环（SD 时钟、屏幕透明）只动其中一个因子 → 收益有限。

三条关键解读

观测含义
update ≈ 22–26主循环被整屏渲染拖慢，实际帧率就是这个数（定时器本应 ~62/s）
命中率 ≈ 87%恰等于 7/8——即"一次读 8 行、首行未命中后 7 行命中"。跨帧复用几乎为零，每帧开头所有带都得重读
reload=0 却 1.7MB/s瓦片集合没变（未跨瓦片边界），但仍在狂读——证明在为"几像素的容器移动"付整屏重读的全价
把 lv_fs_read 临时换成 lv_memset（不读 SD）可作为判决实验：若帧率大涨则 SD 占大头。 但本案数据显示 SD 仅约占帧时间 1/4，大头是整屏 CPU 渲染。

逐项验证常见手段，用"FPS + RTT"对照前后数据，结果如下：

已实施改动 ①：SDIO 提速与硬件诊断

时钟 24MHz → 48MHz（4-bit 并行）

// at32_sdio.c  sd_init() —— 提升 SDIO 时钟
clkdiv = system_core_clock / 48000000;   // 原为 / 24000000
if (clkdiv >= 2) clkdiv -= 2;
sdio_clock_set(clkdiv);   // sdio_ck = ahbclk / (clkdiv+2) = 288 / 6 = 48 MHz

48MHz × 4-bit = 24MB/s 理论上限，而实际只需 1.7MB/s，总线远未跑满，所以提速收益有限。

硬件诊断打印（确认实际位宽/时钟）

// HAL_SD_CARD.cpp  SD_Init() 成功分支
uint32_t clkdiv = ((uint32_t)SDIOx->clkctrl_bit.clkdiv_h << 8)
                  | (uint32_t)SDIOx->clkctrl_bit.clkdiv_l;
uint32_t sdioMHz = (system_core_clock / (clkdiv + 2)) / 1000000;
CONFIG_DEBUG_SERIAL.printf(

    (unsigned)SDIOx->clkctrl_bit.busws, (unsigned)clkdiv, (unsigned)sdioMHz);

已实施改动 ②：周期性 recenter（宏开关，默认关闭）

思路：两次 recenter 之间地图容器完全静止，箭头每帧在静止地图上平滑移动（只重绘箭头小区域， 像设备信息界面一样廉价）；当箭头偏离屏幕中心超过阈值（或瓦片重载/首帧）才整屏 recenter 一次。

已做成宏开关，最终固件默认 OFF（原始实时渲染）。 因为 recenter 虽能把帧率拉到 50–60，但地图会周期性"跳动"，观感不被接受。 开关在 Config.h，改一个宏即可在"高帧率"与"平滑滚动"之间切换，无需改其它代码。

开关与阈值（Config.h）

// 1 = 周期性 recenter（高帧率，地图周期跳动）
// 0 = 原始实时渲染（平滑滚动，每帧整屏重绘，帧率低）—— 当前出厂默认
#define CONFIG_LIVE_MAP_RECENTER_ENABLE   0

// 仅 ENABLE=1 时生效：越大帧率越高、箭头偏离中心越远（须 < 120）
#define CONFIG_LIVE_MAP_RECENTER_MARGIN   48

核心逻辑（LiveMap.cpp · MapTileContUpdate）

lv_coord_t centeredX = baseX - offset.x;   // 让箭头回到屏幕中心所需的容器位置
lv_coord_t centeredY = baseY - offset.y;

bool recenter = forceRecenter || !priv.hasContPos;
if (!recenter) {
    lv_coord_t driftX = priv.appliedContX - centeredX;   // 箭头偏离中心的距离
    lv_coord_t driftY = priv.appliedContY - centeredY;
    if (driftX < 0) driftX = -driftX;
    if (driftY < 0) driftY = -driftY;
    if (driftX > CONFIG_LIVE_MAP_RECENTER_MARGIN ||
        driftY > CONFIG_LIVE_MAP_RECENTER_MARGIN)
        recenter = true;
}
if (recenter) {
    priv.appliedContX = centeredX;
    priv.appliedContY = centeredY;
    priv.hasContPos = true;
}
View.SetMapContPos(priv.appliedContX, priv.appliedContY);  // 位置不变则零重绘
1
平时 — 容器静止，SetMapContPos 传入不变值 → 缓存命中 → 零整屏重绘
2
每帧 — 箭头在静止地图上平滑移动，只重绘箭头小区域 → 廉价，冲 60 FPS
3
偏离 > 48px / 瓦片重载 — 整屏 recenter 一次，箭头回中心
代价（取舍）：地图会周期性"跳一下"而非连续滚动（主流 GPS 表的做法）。 若不喜欢该观感，可调大阈值减少跳动频率，或考虑"中心死区平滑跟随"折中方案（见后续方向）。

帧率天花板与屏幕尺寸

为什么"刷屏速度"提不动了

显示走 SPI1，时钟链：AHB 288MHz → APB2 ÷2 = 144MHz → SPI 最小分频 ÷2 = SCK ≈ 72MHz， 已压在 ST7789 面板写入额定上限（~62–80MHz）。再快要么分频链给不出，要么超规格花屏。整屏传输地板基本定死： 

240×320×2 = 153600 字节 × 8 bit ÷ 72MHz ≈ 17.1 ms   // 整屏 SPI 传输（已接近极限）

若硬件换成 240×240 屏（用平滑滚动旧程序）

240×320（当前）240×240
整屏像素7680057600（−25%）
SPI 传输地板~17.1 ms~12.8 ms
整屏渲染 / SD 读取基准同比 −25%
平滑滚动帧率~25 FPS~32–36 FPS
换屏 25% 少像素 ≈ +33% 帧率，但仍到不了 60：本质没改变"每帧整屏"。 且物理面板是 240×320，软件需改 CONFIG_SCREEN_VER_RES，并修正 ST7789 240×240 变体的 rowstart/colstart 显示偏移；缓冲变小可省 ~38KB RAM（可回补给瓦片行缓存）。
对比两种"减面积"的力度：换屏 −25%，而 recenter 把常态帧从 76800 像素降到约 1600 像素（箭头小区域）， 降约 98%——recenter 远比换屏划算。

增大 LVGL 堆 / 瓦片缓存为什么不是出路

常见的直觉是"瓦片放不进 72KB 堆才慢，把堆调大就好"。但已验证（lv_img_decoder.c:444-447）： LVGL 文件解码器对真彩色图本就逐行流式，无论堆多大都不整图载入。所以：

手段占用 RAM预期收益
增大 LV_MEM_SIZELVGL 堆0——不碰瓦片流式，纯浪费 RAM
增大行缓存 XTRACK_IMG_LINE_CACHE_CNT独立 static 数组跨帧复用 → SD 读取下降，约 +3~5 帧
自定义"整图解码器"（直接给 img_data）专用池，4×128KB=512KB去 SD + 部分渲染开销，约到 ~35 帧
关键：实测 lineReadKB ≈ 1.7MB/s、约 26 次未命中/帧，SD 读取仅占帧时间约 1/4。 大头是整屏 CPU 渲染（~20–30ms）+ SPI 传输（~17ms），而 RAM 一点都碰不到这两块。因此即便把 SD 读取彻底消灭， 平滑（OFF）模式也只能从 ~25 升到 ~30–35 FPS，到不了 60——这是无 GPU + 72MHz SPI 下平滑全屏滚动的物理天花板。

CPU 渲染提速（不换芯片）

既然平滑模式的地板是 CPU 逐像素软件合成（无 2D 加速器），不换芯片就只能从两个方向榨： 降每像素/每对象的计算量 和 让 CPU 跑得更快。先说一个已确认的事实： 编译优化已是 -O3（Optim Level 3），这块基本榨干，不是杠杆。

已发现一个未生效的优化：LV_ATTRIBUTE_FAST_MEM 指向 RAMCODE 段， 但分散加载文件 X-Track.sct 没给 RAMCODE 设独立 RAM 执行区—— 所以 LVGL 标记为"快速内存"的热点绘制函数实际仍从内部 Flash 执行，该优化形同虚设。
手段做法预期收益代价 / 风险
热点绘制函数入 RAM 执行在 .sct 给 RAMCODE 段开 RAM 执行区（Flash 载入、RAM 运行），blend/fill 内循环跑在零等待 RAM内循环提速改分散加载 + 启动拷贝，占几 KB RAM；收益取决于 Flash 缓存现状
预渲染多方向箭头用 24–72 张定角度箭头图替代 lv_img_set_angle 实时旋转（逐像素反变换 + 双线性采样）去掉旋转开销占 Flash 存图；角度量化
轨迹线降本宽 5→2~3、line_rounded=false、必要时关该线抗锯齿去掉粗抗锯齿线的覆盖率计算线条略糙
箭头关抗锯齿lv_img_set_antialias(img, false)旋转采样开销减半边缘略糙
叠加面板不透明sportInfo / zoom 的 bg_opa → LV_OPA_COVER跳过对每帧重绘地图的 alpha 混合视觉变实
ARM-2D 软件加速LV_USE_GPU_ARM2D=1 + 集成库优化的 2D blend 例程M4 无 Helium，收益有限
这些都属于"降每像素/对象成本"。叠加起来平滑（OFF）模式大约从 ~25 提升到 ~35–40 FPS， 对滚动顺滑度是明显改善，且多数无 UX 代价；但仍到不了 60—— 60 只能靠降面积（recenter / 小屏）。

🎯
周期性 recenter（宏开关·默认关）

箭头在中心小框内移动时地图不动，出框才平滑跟随而非硬跳。比 recenter 柔和。待实现。
⚙️
16 位 DMA 传输
DMA 半字 + SPI 16 位帧，逼近 SPI 理论传输上限。仅对 recenter 帧有意义，优先级低。
🗂️
增大瓦片行缓存
让整帧瓦片行常驻，静止时 SD 读取趋近 0。受 RAM 约束，配合换屏省出的 RAM 更可行。
🧵
轨迹线 / 箭头降本
轨迹线变细去圆角、预渲染多方向箭头代替实时旋转。降每像素成本，幅度有限。

结论

导航界面慢，不是显示硬件或 lv_obj_set_pos 的问题，而是"每帧整屏重绘"这件事本身。 整屏一帧的渲染、SD 读取、SPI 传输三块成本都正比于像素数量；SD 提速、关闭屏幕透明等局部优化绕不开 "每帧发一整屏"的前提，因此只涨一两帧。
唯一能上 50–60 的结构性突破是让大多数帧不再整屏重绘（周期性 recenter，把常态帧面积降约 98%）。 但它的代价是地图周期性跳动，观感不被接受，因此做成宏开关、最终默认关闭，保留平滑滚动。
已排除的方向：增大 LV_MEM_SIZE 无效——解码器对真彩色图本就逐行流式，不整图载入； 且实测 SD 读取仅占帧时间约 1/4。靠 RAM/缓存最多把平滑模式从 ~25 推到 ~30–35 FPS，到不了 60。
验证方式：烧录后看 RTT 的 update（≈帧率）与 lineReadKB。 平移帧若冲到 50–60、lineReadKB 大幅下降，即实锤瓶颈是"每帧整屏"且已被绕过。
