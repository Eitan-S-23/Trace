## 一、问题定义(为什么做)

30km/h @ zoom16(分辨率 2.14 m/px)下,画面每秒仅移动 **3.9 像素**。当前渲染以
1px 为最小步进 → 地图每 ~256ms 跳 1 格。离散跳变无运动模糊,人眼对慢速边缘跳动
最敏感,感知为"颤动/卡顿"。**这不是帧率问题**(引擎空闲 84%,提高刷新率只产生
重复帧),是**显示粒度问题**。唯一解 = 亚像素渲染:让地图能停在"第 N.x 像素"处。

量化目标:运动呈现从 `1px @ 4Hz` 细化到 `1/8px @ ~31Hz`,步进幅度缩小 8 倍,
30km/h 低速滚动目视连续无跳格。

## 二、三条技术路线与判决记录

| 路线 | 原理 | 判决 | 证据 |
|---|---|---|---|
| 一:LVGL 变换 | img zoom=257 强制双线性变换路径,pivot 整数移动编码亚像素平移(平移量=pivot/257) | **否决** | 实测 ~70ms/帧(预估 13 的 5 倍),refrMs 1030/s 吃满,帧率反降 12FPS(commit b28a032,坑 10) |
| 二:独立显示缓冲 | snapshotBuf(源)→ 每帧重采样到第二块 240×320×2=150KB 显示缓冲 | **否决** | RAM 不足:可挤出 ~93KB(draw buffer 单缓冲 77+行缓存 16),距 150KB 差 57KB;且单缓冲牺牲 SPI 并行 |
| **三:decoder 层相位混合** | 自定义 lv_img_decoder:LVGL 逐行取快照时,由 read_line 现场输出"行 y 与 y+1 按相位加权"的混合行 | **已终止** | 阶段 0 clean retry 实测:read_line memcpy A=26.10ms/帧(绿灯边缘);固定半像素双线性 B=101.81ms/帧,update=9.97,refrMs=1015.3ms/s,>35ms 终止线 |

路线三的代价:快照放弃 `SRC_VARIABLE` 直拷(~3ms/帧),回到逐行 decoder 调用链。
**纸面成本区间(诚实版,勿再单点乐观):净增 9~16ms/帧**,构成:
- 调用链+blend 固定部分:历史实测锚点——瓦片时代 640 次/帧 read_line 全路径
  ≈20ms/帧,本方案 320 次(单 img)≈ **10ms**;
- 混合增量:滑动窗口 2 次 mix/px(见 3.3)≈20-25cyc × 76800px ≈ **5-7ms**(RAMCODE 内);
  AC5 优化保守/不进 RAMCODE 时可 ×1.5-2。
总帧预期 24~31ms——**贴着 31Hz 预算线,探针闸门(见阶段 0)是硬保障**。

**阶段 0 实测结论(2026-07-10 clean retry,设备压测 300/400km/h):路线三终止。**
- A(read_line 仅 memcpy):前 30 条 RTT 平均 `refrMs=647.2ms/s`,`fps=24.80`,总帧时间 `26.10ms`,`update=33.53`。调用链成本已把 19ms/帧基线推到 28ms 绿灯线边缘。
- B(固定半像素双线性+REV16):前 30 条 RTT 平均 `refrMs=1015.3ms/s`,`fps=9.97`,总帧时间 `101.81ms`,`update=9.97`。远超 `>35ms` 终止线,不进入阶段 1。
- 数据文件:`.claude/subpixel_probe_a_clean_retry_rtt_2.log`、`.claude/subpixel_stage1_decoder_rtt_0306.log`。本轮按 J-Link hygiene 重测:烧录后重查 `_SEGGER_RTT`、验证 `SEGGER RTT` 签名、按环形 down channel 发送 `gpsreset`/`livemap`、单 logger 采集。探针代码与压测速度已清理/恢复。

## 三、路线三详细设计

### 3.1 数据流

```
插值(8.8 定点,已留存) ──相位 fx,fy(0..255)──┐
                                             ▼
snapshotBuf(源,整px对齐,256×320) ──► 自定义 decoder read_line(y):
                                       out[x] = 双线性( src[y][x..x+1], src[y+1][x..x+1], fx, fy )
                                             ▼
                                     LVGL sw blend(与瓦片 img 同路径,已验证多年)──► draw buffer ──► SPI
```

### 3.2 自定义 decoder 要点

- 注册:`lv_img_decoder_create()` 新建 decoder,`info_cb/open_cb/read_line_cb/close_cb`。
  识别方式:快照专用 `lv_img_dsc_t` 的 `header.cf` 用 `LV_IMG_CF_USER_ENCODED_0`
  (LVGL 保留的用户格式,built-in decoder 不认它,自动路由到本 decoder)。
  **关键细节(漏掉即白屏):`info_cb` 必须把输出 `header.cf` 声明为
  `LV_IMG_CF_TRUE_COLOR`**——USER_ENCODED_0 仅用于 decoder 路由,绘制引擎
  只认标准格式;open 置 `img_data=NULL` 强制走 read_line。其他图片不受影响。
- read_line(x, y, len, buf):
  - 源行指针 `r0 = snapshotBuf + y*256`,`r1 = (y<319) ? r0+256 : r0`(**最底行 clamp**,
    半像素误差仅最后一行,不可见);
  - 水平 x+1 越界:快照 256 宽 > 视口 240,margin 天然覆盖(x+len+1 ≤ 256 恒成立,
    因 LVGL 请求的 len 是可见交集宽 ≤240,x 由 img 相对位置决定落在 [0,16] 网格偏差内)。
    仍需防御式 clamp。
- 相位传递:模块级 `static uint8_t snapPhaseX, snapPhaseY;`,由 `Snapshot_Update`
  每周期写入(相位 = `dispMapXFp & 0xFF` 折算到 1/8px 粒度)。

### 3.3 RGB565 双线性定点混合(内循环,决定成败)

```c
/* RGB565 双通道位并行加权:R|B 同乘,G 单独,权重 w∈[0,32](5bit,>>5 归一) */
static inline uint16_t rgb565_mix(uint16_t a, uint16_t b, uint16_t w /*0..32*/)
{
    uint32_t arb = a & 0xF81F, agr = a & 0x07E0;
    uint32_t brb = b & 0xF81F, bgr = b & 0x07E0;
    uint32_t rb = (arb + (((brb - arb) * w) >> 5)) & 0xF81F;
    uint32_t g  = (agr + (((bgr - agr) * w) >> 5)) & 0x07E0;
    return (uint16_t)(rb | g);
}
/* 滑动窗口:先对列做垂直 mix(结果 v[x]),相邻像素的 v[x+1] 即下一列的
 * 垂直 mix,可复用——每输出像素净成本 = 1 次垂直 mix + 1 次水平 mix
 * = 2 次 mix ≈ 20-25 cycles。伪码:
 *   vPrev = mix(r0[x0], r1[x0], wy);
 *   for x: vCur = mix(r0[x+1], r1[x+1], wy);
 *          out[x] = mix(vPrev, vCur, wx); vPrev = vCur;  */
```
**默认要求(非备选)**:read_line 内循环函数加 `LV_ATTRIBUTE_FAST_MEM` 进
RAMCODE 段(AC5 优化保守 + Flash 取指等待,不进 RAM 成本可翻倍)。
优化备选(若探针仍超标):相位量化 1/8 权重查表;只垂直混合(折衷,慎用)。

**已查实:`LV_COLOR_16_SWAP = 1`(lv_conf.h:34)**——快照像素是字节交换
格式,G 通道被拆成两段非相邻位,上述掩码**不能直接用**(症状:绿色泛滥)。
定案:mix 前后用 `__REV16`(M4 单周期字节交换指令)夹心——读入 swap 回
标准位序 → 混合 → 输出 swap 回去,每像素 +~3 cycles(已计入成本区间上限)。
模拟器同为 SWAP=1,颜色验证可在模拟器完成。

### 3.4 步进与重绘节流

- 插值最小步进:1/8px(`interpMinStep=32`,SUBPIXEL v2 分支——**注意坑:此值曾在
  SUBPIXEL=0 时误用导致 22 帧/s 无效重绘,分支语义必须保持**);
- 重绘门槛:相位或整数坐标变化 ≥1/8px 才 invalidate(30km/h → ~31 次/s 重绘);
- 高速自动降级:像素速度 >40px/s 时(1px 步进本身已 >40Hz,肉眼连续)相位强制 0、
  走纯拷贝分支,压测帧率不回退。

### 3.5 视觉效果的两个次级效应(阶段 2 目视验收时的判读依据)

- **运动中轻微模糊 = 特性而非缺陷**:半像素相位下双线性混合使边缘对比度略降,
  等效"运动模糊"——正是电影 24fps 平滑感的来源;静止时相位吸附到整数(插值
  SNAP 机制),画面完全锐利。判读:静止锐利+运动微糊 = 正常;静止也糊 = 相位
  没有吸附归零,查插值收敛。
- **缓动不匀速可能显形**:指数缓动(剩余 1/DIV)的步进先快后慢,1px 粒度下
  不可见,1/8px 粒度下可能呈现轻微"呼吸节奏"。若阶段 2 目视有此感,备选
  改匀速插值(按 GPS 周期位移均分到各 tick),不影响本计划其余部分。

## 四、实施阶段(强制顺序,每阶段有判据)

### 阶段 0:性能探针(半天,不达标即终止)

**目的**:拿到"decoder 逐行调用链"与"相位混合"在本机的真实每帧成本(坑 10 纪律)。本阶段已完成,结果触发终止。

0. **测量工况统一用压测速度**(`SIM_SPEED_MIN/MAX_KPH` 临时 300/400,测完恢复
   50/80):重绘满频(~31 次/s)使"总帧时间 = refrMs/每秒整屏数"统计稳定;
   30km/h 每秒仅 4 帧,噪声大不可作判据工况。混合成本与滚动速度无关,
   压测数据对低速同样成立。
1. 注册探针 decoder(识别 USER_ENCODED_0),read_line 仅 memcpy(无混合);
   快照 dsc 的 cf(onViewLoad 中 snapshotDsc.header.cf)临时改 USER_ENCODED_0
   → 烧录 → 复位+gpsreset+livemap → RTT 采集 30s。
   **读数 A = 总帧时间**(纯调用链成本;对照基线:压测下直拷 ~19ms/帧)。
2. read_line 加满血双线性(固定相位 128)→ 烧录 → 采集。**读数 B = 混合总帧时间**。
3. **判据(锚定体验预算=31Hz 周期,写死,中途不得修改——防"移动球门")**:
   以读数 B 计算**总帧时间 = refrMs/每秒整屏数**(30km/h 工况):
   - 总帧 ≤28ms → 绿灯,进入阶段 1;
   - 28~35ms → 启用 3.3 优化备选后**复测一次**,≤28ms 进阶段 1,否则终止;
   - >35ms → **终止**,结论与数据归档到复盘手册坑 10 下,路线三关闭,
     平滑度课题标记"本硬件无解,待矢量化/换片"。
4. 探针代码全程用 `TEMP-PROBE` 注释标记,阶段结束后删除或转正。

### 阶段 1:静态相位正确性(半天)

- 固定相位 (128,128)(半像素),模拟器编译;
- 验收:①截图与相位 (0,0) 对照,内容整体偏移半像素(边缘线条变柔);
  ②无花屏/错位/裁剪;③16/17/18 级(√2 放大路径)各截一张确认与相位混合共存正确。

### 阶段 2:动态接入(1 天)

- `Snapshot_Update` 传递真实相位;插值 SUBPIXEL v2 分支(1/8px 步进+吸附 32);
- 重绘门槛与高速降级(3.4);
- 设备验收:①`gpsreset+livemap` 日常工况 RTT:refrMs ≤ 500/s、无 update 掉速;
  ②30km/h(临时 SIM_SPEED 25/35)**目视滚动连续无跳格**(用户确认);
  ③400km/h 压测:≥24FPS 不回退(高速降级生效);
  ④快照 dump 转 PNG:混合输出无色偏(rgb565_mix 掩码错误的指纹是绿色泛滥)。

### 阶段 3:收尾(半天)

- 边界回归:zoom 快速拨动、gpsreset 大跳变、出界白图、RouteSelect 进出页;
- 删 TEMP-PROBE、更新两份手册与 PLAN 状态、提交(实验纪律:一个 commit 一个结论)。
