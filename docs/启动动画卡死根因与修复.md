# 开机动画卡死：根因分析与修复说明

> 适用范围：AT32F435（X-Track 派生固件，ARM Compiler 5 / AC5，LVGL v8，240×320 竖屏）。
> 结论先行：旧版"赛博电路"开机动画用了 **`LV_EVENT_DRAW_POST` 自绘 + 每帧自定义动画 + 强制全屏重绘** 的架构，
> 在本 32 位设备上会**破坏 LVGL 内存**导致跳转仪表盘前 HardFault/重启；新版改用 **纯真实对象 + `anim_timeline`**
> 的架构后稳定运行。下文给出现象、排查、根因、机制与规范。

---

## 1. 现象

- 设备烧录后：开机动画播放，**在跳转仪表盘前卡死**（HardFault 停机，或看门狗无限重启）。
- 同一套代码在 **Windows LVGL 模拟器里完全正常**。
- 首个 RTT 崩溃 dump 的特征（多次复现，值稳定）：
  - `R0 = R1 = 0x042120D1`，读 `[R0+4]` 触发**精确数据总线错误**（`CFSR: PRECISERR`，`BFAR=0x042120D5`）。
  - 调用栈（用匹配的 axf 翻译）：`Replace("Pages/Dialplate") → … → DialplateView::Create → Value_Create →
    lv_label_set_text → lv_obj_refresh_style → lv_obj_invalidate → lv_obj_get_screen → lv_obj_get_parent` 崩。
  - 即：创建仪表盘第一个标签、刷新样式时，沿**父对象链**向上找屏幕，读到某个对象的 `parent` 字段是**野值** `0x042120D1`。

关键观察：`0x042120D1` 拆成小端两个 16 位 = `0x20D1`、`0x0421`，**正好像两个 RGB565 暗青/暗蓝像素值**——指向"渲染产生的数据被写进了对象指针字段"。

---

## 2. 排查过程（被逐一排除的方向）

为避免凭空猜测，做了大量对照实验，**排除**了以下方向：

1. **不是 LVGL 内存池耗尽 / "lv_obj_create 吃 85KB"。**
   把模拟器改成与设备**完全一致的内存模型**（`LV_MEM_CUSTOM=0` 的 96/128KB TLSF 池、`operator new` 也走 `lv_mem_alloc`），
   实测跑完 启动→仪表盘创建 全程：峰值仅约 **38KB / 池容量**，`lv_mem_test()` 始终 `integ=OK`，无泄漏无耗尽。
   设备 32 位指针只会更省。**上一阶段一度怀疑的"池耗尽/85KB"是诊断探针自己每帧遍历堆、拖垮主循环触发 10 秒看门狗
   造成的假象**（`StartupView::Create` 实测只占 +136 字节）。

2. **不是绘制缓冲"半屏 vs 全屏"。** 曾把设备绘制缓冲从 2×半屏改成 1×全屏，崩溃**完全照旧**，已否定。

3. **不是"画的内容"。** 把旧动画逐步削减：去掉阴影/渐变/抗锯齿圆头线、去掉每帧 anim、去掉 `lv_draw_label`……
   **每一种削减都仍然崩**，只是崩溃点在"创建期 vs 渲染期"之间漂移。说明元凶**不是某个具体绘制调用**。

4. **不是页面缓存/生命周期。** 旧的、能正常工作的 f58e4ed 启动页**同样**设置了 `SetCustomCacheEnable(false)`；
   改动缓存/提前删除场景并不能解决问题。

5. **不是 ResourcePool 字体/图片缺失、不是 Account/DataCenter、不是 lv_allocator。** 逐一核对均正常。

最终用 **A/B 对照** 锁定根因：
- 仅把 `StartUp/` 三个文件回退到 **f58e4ed（真实对象盒子+标签 + `anim_timeline`）→ 设备恢复正常进入仪表盘**。
- 换回新版那套"自绘架构"→ 立刻复发。
- 二者差异只在 **启动页的实现架构**，与其它代码无关。

---

## 3. 根本原因（已确证：架构层面）

**旧版开机动画（崩溃版 `StartUpView`）的架构本身是元凶**，具体由以下几点叠加构成：

```
旧版（崩溃）架构：
  1) 建一个全屏 scene 对象；
  2) 给它挂 LV_EVENT_DRAW_POST 事件回调，在回调里用 lv_draw_rect / lv_draw_line /
     lv_draw_label 直接“自绘”整幅电路（含渐变、阴影、抗锯齿圆头线、芯片文字）；
  3) 用自定义 lv_anim 每帧改 flow_phase / brand_phase；
  4) 用一个 33ms lv_timer 每帧 lv_obj_invalidate(scene)，强制整屏重绘 → 上面的自绘回调每帧执行。
```

问题出在 **第 (2) 点：在事件回调里直接调用 `lv_draw_*` 自绘**。这些底层绘制函数（尤其带**阴影、渐变、抗锯齿圆头线**时）
会让 LVGL 的软件渲染器在渲染过程中**从 LVGL 堆临时申请 mask / 混合 / 渐变缓冲**，并按"当前绘制区域 / 绘制缓冲几何"
做大量指针运算与逐像素写入。这条"自绘 + 软件渲染器临时缓冲"的路径，在本设备的具体条件下（见第 4 节）会发生
**几字节级的越界写**，把相邻的一个 LVGL 堆块 / 对象的指针字段覆写成**渲染数据**（即那个像素值 `0x042120D1`）。

被写坏的指针字段是"潜伏"的：当时那块内存可能是空闲的或属于启动页对象；等启动页销毁、仪表盘对象复用到同一块内存、
随后访问它的 `parent` 指针时，才解引用这个野值而总线错误——所以崩溃**表现在"跳转仪表盘"阶段**，但**起因在启动动画的自绘**。

> 诚实边界：**"自绘架构是元凶"经 A/B 对照确证**；但**那条越界写的确切机器指令未能 100% 定位**——
> 本机无硬件数据观察点（DWT watchpoint）、x64 模拟器根本不复现（见下）、设备上崩溃点每次构建都漂移（典型内存破坏 Heisenbug）。
> "渲染器临时缓冲越界写"是与全部证据一致的**强推断**，而非逐指令证明。

---

## 4. 为什么"只在设备崩、模拟器永远正常"

这正是它极难排查的原因。设备与模拟器在**同一份 C 代码、同一份 LVGL** 下，仍有几处关键差异，使同一处越界
在两边后果完全不同：

| 维度 | 设备（AT32F435 / AC5 / ARM32） | Windows 模拟器（MSVC / x64） |
|---|---|---|
| 指针/对齐 | 4 字节指针、TLSF 块按 **4 字节**对齐，**几乎无空隙** | 8 字节指针、堆块按 **8 字节**对齐，**有对齐空隙** |
| 分配器 | LVGL 自带 TLSF（`LV_MEM_CUSTOM=0`） | 标准 `malloc`（`LV_MEM_CUSTOM=1`），块尾有余量 |
| 绘制缓冲 | 半屏双缓冲 + SPI DMA 刷屏 | 全屏单缓冲 + GDI |

**后果**：那条越界写若只多写几字节，在 **x64**（8 字节对齐 + malloc 余量）里**落进对齐空隙/块尾余量，无害**——
所以模拟器跑得好好的；而在 **32 位设备**（4 字节对齐、无空隙）里**正好砸到下一个块的块头或某个活动对象的指针字段** → 崩。
实测：把模拟器编成 **x86（32 位）** 后**也会崩**，与设备一致——进一步印证"32 位特有的越界，被 x64 的对齐空隙掩盖了"。

补充：设备上 LVGL 的断言/日志在 `ARDUINO` 下是关闭的（`LV_USE_LOG=0`），所以越界发生时**毫无报错**，带着坏指针
一路跑到很后面才崩——这也是为什么崩溃点看起来在仪表盘、且每次构建都漂移。

## 5. 新版为什么安全

新版开机动画**彻底不碰**那条危险路径，改用**与旧的、已验证稳定的启动页同一套机制**：

```
新版（稳定）架构：
  • 只用真实 lv_obj 对象拼出画面：
      - 走线 / 引脚 / 端点 / 流动光条 = 一个个矩形对象（lv_obj，带 bg_color 样式）；
      - 芯片 = 圆角 + 边框样式的 lv_obj（圆角/边框样式本身是安全的，旧版能跑的盒子就带边框）；
      - “Loading” / “E-Track” = 标签对象（lv_label）/ 像素格矩形对象。
  • 动画只用 anim_timeline + lv_anim：
      - 光条流动 = 自定义 anim 回调，但回调里**只调用 lv_obj_set_pos / lv_obj_set_size 移动和缩放对象**
        （不画任何东西）；
      - 像素艺术字“逐列点亮” = 把像素格放进一个**裁剪容器**，用 anim 推进容器 width 0→满，靠裁剪实现从左到右揭示。
  • 不挂 LV_EVENT_DRAW_POST、不调用 lv_draw_*、没有强制每帧整屏 invalidate 的刷新定时器。
```

关键区别：**渲染交给 LVGL 标准的对象绘制流程**（经过充分验证、不在我们的事件回调里临时乱申请缓冲），
我们只负责"摆对象 + 用动画改对象的位置/尺寸"。`lv_obj_set_pos/size` 这类对象 API 是安全的；危险的从来是
"在 `LV_EVENT_DRAW_POST` 里自己调 `lv_draw_*` 画图"。因此新版在设备上稳定。

> 副带修正：`agencyb_36` 是为 "X-TRACK" 定制的子集字体、**不含字母 'E'**，直接用它显示 "E-TRACK" 会出现豆腐框；
> 故 E-Track 采用**像素字模拼矩形对象**实现，既安全又有艺术字效果。

## 6. 经验教训 / 后续规范

1. **嵌入式 LVGL 上不要在 `LV_EVENT_DRAW_POST`（或任何绘制事件）里直接 `lv_draw_*` 自绘复杂图形**
   （尤其阴影/渐变/抗锯齿）。要静态装饰用**真实对象**或**预生成图片**（如仪表盘皮肤 `img_src_dialplate_skin`，
   它是 Flash 常量图、稳定）；要动效用 **`anim_timeline` 改对象属性**。
2. **"模拟器正常"不等于"设备正常"**：x64/64 位对齐会掩盖 32 位上的越界。排查这类问题，可把模拟器编成 **x86**
   或用带 redzone 的分配器复现。
3. **崩溃 dump 的翻译要用"与所烧固件地址匹配的 axf"**：`build_f435.ps1` 重建后用
   `arm-none-eabi-addr2line -e Objects/X-Track.axf -a -f <地址列表>`。注意 dump 里的 `Build: ...` 时间戳是
   `HAL_FaultHandle.cpp` 的 `__TIME__`（其上次编译时间），增量构建不变，**不能据此判断烧的是哪一版**。
4. **内存破坏型 Heisenbug 的崩溃点会随改动漂移**，不要被"换个地方崩"误导成"修了一部分"；要抓**架构层面的对照**。
